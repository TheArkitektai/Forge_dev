# Arkitekt Forge: Implementation Spec V2 (Interaction Depth)

> **Purpose**: This spec addresses critical interaction depth gaps in the current demo build. The screens exist and render content, but the demo lacks the *story continuity*, *human in the loop feedback*, *agent rework simulation*, *design artifact lifecycle*, and *live output preview* that make the product believable as a governed AI delivery platform.
>
> **What this spec does NOT cover**: Screen layouts, basic routing, persona lens switching, sidebar navigation, and data foundation are already built and working. This spec is purely about adding the missing interaction patterns on top of the existing codebase.
>
> **Formatting rule**: Never use dashes (hyphens, en dashes, em dashes) in any UI text, labels, or copy. Use colons, commas, or rewrite instead.
>
> **Build order**: Follow the phases in sequence. Each phase builds on the previous one.

---

## PHASE 1: Story Detail Panel (Expandable Drawer)

### Problem
When you click a story card in Delivery Flow, it selects the story and shows persona focus text in the right rail. But there is no way to:
- See the full story detail (description, acceptance criteria, history, comments)
- Give feedback on the story
- See what the agent has produced for this story in the current phase
- See the story's journey through phases (timeline of transitions)

### What to Build

#### 1.2 Phase Specific Agent Output Sections

Each phase should show what the agent has produced. This is pre scripted seed content (not real AI).

**Plan phase output:**
- Brief document: structured text showing problem statement, scope, stakeholders, success metrics
- Agent badge: "Generated by Planning Agent · Reviewed by Maha Noor"
- Status: "Awaiting human approval" or "Approved"

**Design phase output:**
- Design document: component diagram (use a styled card grid showing components with arrows between them, not a real diagram tool)
- User flow: 4 to 6 step flow rendered as horizontal connected cards (step 1 → step 2 → step 3)
- Wireframe placeholder: styled card with description (e.g., "Citizen authentication screen: national ID input, OTP verification, biometric fallback")
- Architecture impact summary: which layers are affected (links to Architecture screen)
- Agent badge: "Generated by Design Agent · Pending architect review"

**Develop phase output:**
- Code summary: key files changed, lines added/removed (use GitHub data)
- Test coverage: percentage bar
- Agent badge: "Generated by Development Agent · Code review pending"

**Test phase output:**
- Test results: pass/fail counts with a simple bar
- Security scan: status badge (passed/warnings)
- Agent badge: "Generated by QA Agent · Release gate pending"

**Ship phase output:**
- Deployment checklist: items with checkmarks
- Release notes: auto generated text
- Agent badge: "Generated by Release Agent · Final approval required"

## PHASE 3: Consistent Human in the Loop Pattern

### Problem
The platform claims "zero autonomous action" but the demo only has forward movement (advance story). There is no visible pattern of: human reviews → rejects → agent reworks → human reviews again. This is the core value proposition.

### What to Build

#### 3.1 Rework Simulation Engine

Add to `context.tsx` a reusable rework simulation function:

```typescript
const simulateAgentRework = useCallback((
  entityType: "story" | "design_artifact",
  entityId: string,
  rejectionReason: string
) => {
  // Step 1: Immediate state change
  // Update entity status to "reworking"
  
  // Step 2: Show AI agent status
  setAiAgentStatus("thinking");
  addNotification({
    title: `Agent reviewing rejection feedback`,
    body: `Analyzing: "${rejectionReason.slice(0, 50)}..."`,
    type: "info"
  });
  
  // Step 3: After 2s, show "compiling"
  setTimeout(() => {
    setAiAgentStatus("compiling");
  }, 2000);
  
  // Step 4: After 4s, show "generating" 
  setTimeout(() => {
    setAiAgentStatus("generating");
  }, 4000);
  
  // Step 5: After 6s, complete rework
  setTimeout(() => {
    setAiAgentStatus("complete");
    // Update entity with reworked content
    // Add "[Reworked]" prefix or version increment
    // Add notification: "Rework complete. Ready for review."
    // Add audit trail event
    
    // Step 6: After 2 more seconds, reset agent status
    setTimeout(() => setAiAgentStatus("idle"), 2000);
  }, 6000);
}, []);
```

This creates a visible 6 second simulation where the AI agent panel (already built in Command Center right rail) shows the agent working. The key insight: the `AIAgentPanel` component already has the status machine UI (thinking → compiling → generating → complete). We just need to trigger it from reject actions.

#### 3.2 Rejection Flow on Every Screen

**Delivery Flow:**
- Story cards in the kanban show a small "reworked" badge if `story.agentOutputs[phase].status === "reworked"`
- Story detail drawer shows the rework history in the feedback section
- "Reject & Rework" button in drawer triggers `simulateAgentRework`

**Architecture:**
- Design artifact cards show version number and rework badge
- Reject triggers rework simulation, artifact version increments
- After rework, the artifact card pulses briefly (framer motion) to draw attention

**Governance:**
- When rejecting an approval item, if it has an `artifactId`, the design artifact is also updated
- If it has a `storyId`, the story gets a feedback entry
- The approval queue item shows "Rejected → Agent reworking..." during the simulation, then "Reworked · Awaiting re review"

**Context Hub:**
- Memory events should include entries for rejections and reworks
- Add 2 to 3 seed memory events of type "lesson_learned" that reference past rejections (e.g., "API contract v1 was rejected because it exposed PII in query parameters. Agent corrected to use request body. This pattern is now stored for future projects.")

#### 3.3 Rework Status Badges

Create a small reusable component `ReworkBadge.tsx`:

```typescript
// Shows different states
"awaiting_review"  → amber badge: "Awaiting Review"
"reworked"         → blue badge: "Reworked · v{n}" 
"approved"         → green badge: "Approved"
"rejected"         → red badge: "Rejected"
"reworking"        → animated badge with spinner: "Agent Reworking..."
```

Use this badge consistently on:
- Story cards in Delivery Flow kanban
- Design artifact cards in Architecture
- Approval queue items in Governance
- Agent output sections in Story Detail Drawer

## PHASE 5: Cross Screen Story Continuity

### Problem
Each screen works independently. When you approve a design in Architecture, nothing visibly changes in Delivery Flow or Governance. When you reject in Governance, the story in Delivery Flow does not reflect it.

### What to Build

#### 5.1 Shared Story Event Bus

All state changing actions should trigger cascading updates across screens. This is already partially done through the shared context, but needs to be more visible.

**Update `advanceStory` in context:**
When a story advances from Design → Develop:
1. All design artifacts for that story get status "approved" (if not already)
2. Governance queue items related to Design phase get auto resolved
3. A new set of governance items for Develop phase gates are added
4. Output artifacts for the Design phase become available in the Output screen
5. Memory events are added to Context Hub

**Update `rejectStory`:**
When a story is rejected from Design → Plan:
1. Design artifacts for that story get status "rejected"
2. Governance queue items for Design phase get marked "Returned for rework"
