1. Guiding Philosophy

NEW in v2: Subscription First. The subscription and metering system is the very first module built, before any agent execution or governance pipeline. The reason is simple: Forge uses Claude SDK (and potentially Kimi or other LLM providers) for agentic development. Those token costs are not in Uxbert's control. Every API call to an LLM provider costs real money. Without metering, budget caps, and subscription enforcement from Day 1, a single runaway agent execution or a misconfigured tenant could consume thousands of dollars in tokens before anyone notices. Subscription is not a billing feature. It is the financial safety net that makes everything else commercially viable.
Core Rule: Forge is the product. NEOM is the proof. Every line of code is a generic Forge product feature. NEOM configures and deploys that product. A second customer on Day 1 gets the same platform with different configuration.

Architecture v12 Is the Law

The Agentic Forge Architecture v12 document defines nine layers, eighteen principles, twenty two Kanban states, and the complete module system. This plan is a build schedule for that architecture. It does not add, remove, or reinterpret any architectural decision. If a conflict arises between this plan and the v12 document, the v12 document wins.

Prototype as UI Reference, Not Codebase

The arkitekt-forge-ui-refreshv2.0 prototype validates screen layouts, component patterns, persona flows, and interaction models. The production build will reuse its design system (shadcn/ui + Tailwind + Framer Motion) and component naming conventions, but replace all mocked data with real API integration.

Contract First Development

Before building any module, a dedicated session defines its API contract in @forge/contracts. All subsequent sessions implement against those contracts. This prevents interface drift.

2. Subscription and Metering Architecture (Day 1 Priority)

Why Day 1: Your financial model budgets LLM inference at 2% of subscription revenue in Year 1, ramping to 3.5% by Year 5. At $8 per million tokens (Year 1 blended rate), that translates to tight but workable token allocations per tier. Without metering enforced at the API boundary from the very first deployment, there is no way to validate that actual LLM costs stay within the 2% budget. This is not a billing convenience. It is margin protection.

2.1 The Cost Problem

Every agent execution in Forge calls an LLM provider (Claude, Kimi, or via OpenRouter). Uxbert pays per token. The model mix uses Haiku for classifiers (cheapest), Sonnet for reasoning (mid tier), and Opus for complex architecture (most expensive). Costs vary by model, context length, and execution complexity. A single story averages 150,000 tokens, but a complex architecture story with 5 iteration cycles could consume 750,000 tokens or more.

Without controls, the risk profile looks like this:

2.3 Metering Architecture

The metering system intercepts every LLM call at the Execution Service layer (the abstraction that wraps Claude SDK, Kimi, OpenRouter, or any provider). This is the single choke point where all token consumption flows through, regardless of which agent, tool, or workflow initiated the call.

Metering Pipeline (6 Steps)
StepWhat HappensWhere
1. Pre flight CheckBefore any LLM call, the Execution Service queries the tenant's current usage against their allocation. If the tenant has exceeded their hard cap, the call is rejected with a structured error. If within 80%/90% threshold, a warning event is emitted.@forge/subscription (middleware in @forge/agents)
2. Model SelectionThe Model Router selects the cheapest model that meets the quality threshold for this task type. Classifiers go to Haiku. Reasoning goes to Sonnet. Complex architecture goes to Opus. This is the first line of cost control.@forge/agents (Model Router)
3. ExecutionThe LLM call executes. Input tokens and output tokens are counted from the provider response.@forge/agents (Execution Service)
4. Metering EventA metering event is published to Redis Streams with: tenant_id, project_id, story_id, agent_execution_id, model_used, input_tokens, output_tokens, cost_at_provider_rate, timestamp.@forge/events
5. Usage AggregationA BullMQ worker aggregates metering events into tenant usage counters in PostgreSQL. Running totals updated every 30 seconds. Daily, weekly, and monthly rollups for reporting.@forge/subscription (BullMQ worker)
6. Budget EnforcementBudget alerts emitted at configurable thresholds (default 80%, 90%, 100%). Hard cap enforcement kicks in at the configured ceiling. Admin notification via webhook and dashboard.@forge/subscription + @forge/events

2.5 Cost Attribution Hierarchy

Every token consumed is attributed at four levels. This enables usage dashboards at every scope: organization wide, per project, per team, and per individual story execution.

LevelWhat It AnswersConfigurable Budget?
Tenant (Organization)How much is this customer consuming against their subscription allocation?Yes. Hard cap, included allocation, overage rate, alert thresholds.
ProjectWhich project within the tenant is consuming the most? Should this project have its own sub budget?Yes. Optional project level budget (subset of tenant allocation). Configurable by tenant admin.
Team/User GroupWhich team is driving consumption? Is one team consuming disproportionately?Yes. Optional team level budget. Configurable by project lead.
Story/ExecutionWhich specific story or agent execution consumed how many tokens? What model was used?Yes. Per execution token ceiling (architecture v12: Context Budget Manager). Default from workflow template, overridable per story.

2.6 Admin Configuration Console

All subscription and metering parameters are configurable by Uxbert admins (platform level) and by tenant admins (within their allocation). The system ships with sensible defaults from the tier matrix above, but nothing is hardcoded.

What Uxbert Admins Can Configure (Platform Level)
  • Tier definitions: name, included tokens, overage rate, hard cap, max projects, max users, max concurrent agents
  • Subscription duration and renewal terms per tier
  • Grace period duration per tier
  • Data retention period per tier after suspension
  • LLM provider cost rates (updated when provider pricing changes)
  • Model routing rules: which models are available per tier, cost/quality thresholds
  • Global rate limits: max tokens per minute, max concurrent LLM calls per tenant
  • Feature flags: which modules and governance gates are available per tier
  • Trial settings: duration, allocation, feature restrictions
  • Overage behavior: allow overage (bill later) or block at cap
  • Per tenant overrides: any allocation can be overridden for a specific tenant
What Tenant Admins Can Configure (Within Their Allocation)
  • Project budgets: allocate portions of tenant token budget to specific projects
  • Team budgets: allocate portions of project budget to specific teams
  • Per story execution ceilings: max tokens per agent execution
  • Alert recipients: who gets notified at each threshold (email, webhook, dashboard)
  • Model preferences: prefer cheaper models for certain story types
  • Auto scaling rules: automatically request tier upgrade when approaching limits (Enterprise/Sovereign)

2.9 Stripe Integration (or Alternative)

For automated billing, Stripe is the recommended payment processor. The integration handles: subscription creation and management via Stripe Billing, automatic invoice generation, payment method management (card, bank transfer for Enterprise/Sovereign), overage billing as metered usage items on the Stripe subscription, webhook handling for payment success/failure/dispute events, prorated upgrades and downgrades, and tax calculation via Stripe Tax. For Saudi government accounts where Stripe may not apply, the system supports manual invoicing mode where usage is tracked but billing is handled offline through procurement processes.

6. Build Dependency Chain

contracts → db → events → subscription → api → workflow → governance → agents → dashboard → context-hub → connector-framework → tenant-manager → module-registry

Critical path change from v1: @forge/subscription is inserted between @forge/events and @forge/api. The API server middleware enforces tier limits on every request. The agent execution pipeline queries subscription status before every LLM call. Nothing that costs money runs without metering.

Session 05: Subscription and Metering Engine
Billing

Module: @forge/subscription

This is the critical Day 1 module. It must be complete and tested before any agent execution is enabled.

  • Tier Management: CRUD for subscription tiers with all configurable parameters
  • Subscription Lifecycle: State machine (Trial, Active, Grace, Suspended, Terminated, Upgraded, Downgraded)
  • Subscription Expiry: Automated expiry detection via BullMQ scheduled job. Grace period enforcement. Read only mode during grace. Account suspension after grace. Data retention timer.

  • Cost Tracking: Provider cost recording (actual cost to Uxbert per LLM call). Customer cost calculation (included allocation vs overage). Margin calculation per tenant (revenue minus LLM cost minus infrastructure).
  • Overage Handling: Track tokens beyond included allocation. Calculate overage cost at tier specific rate. Generate overage line items for billing.
  • Admin Overrides: Per tenant allocation overrides (any parameter). Temporary budget increases (with expiry). Emergency kill switch (immediately block all LLM calls for a tenant).
  • Stripe Integration: Subscription creation and management. Metered billing for overage. Webhook handling (payment success/failure). Invoice generation. Manual invoicing mode for government accounts.
  • API Endpoints: All endpoints defined in Section 2.8 above.
  • Dashboard Widget Data: Usage gauge, cost breakdown, budget status, subscription status, expiry countdown.
