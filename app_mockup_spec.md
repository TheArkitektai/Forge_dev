# Arkitekt Forge working product mockup specification

## Corrected direction

This mockup must behave like the final product, not like a presentation page. The user should be able to move through a believable Arkitekt Forge application shell and understand how the main capabilities work together inside one product.

The entire experience should revolve around one active program and one active release train so every screen feels connected. The same governed work, the same context, the same architecture, the same evidence, and the same configuration logic must appear across the application in different but coherent ways.

## Final navigation model

The application should use a persistent left navigation and a contextual top bar.

| Navigation area | Screen purpose | What it proves |
| --- | --- | --- |
| Command Center | Cross persona overview of momentum, approvals, risk, and release confidence | The platform is operationally clear and executive ready |
| Delivery Flow | Real six phase work progression, approvals, story detail, and blockers | Arkitekt Forge governs the SDLC end to end |
| Operate | Production monitoring, incident response, service health, and lesson capture | Operations are part of the governed lifecycle |
| Context Hub | Institutional memory, related projects, prior decisions, reusable patterns, and timeline context | Memory is a core operating surface |
| Architecture | Layers, modules, impacted services, design rationale, and dependency changes | Architecture is visible and connected to delivery |
| Governance | Proof chain, control checks, evidence packs, overrides, and release readiness | Trust and auditability are built into the product |
| Config Studio | Workflow builder, policy controls, retention tiers, and module activation | Configurability is native to the platform |

## Shared product spine

Every screen should reference the same active initiative and story set.

| Shared entity | Role in the mockup |
| --- | --- |
| Initiative | National Digital Permits Platform |
| Current release | Release 24.4 governed launch |
| Featured story | Citizen Identity Verification orchestration |
| Active architecture concern | Identity service, policy engine, audit graph, and context compiler integration |
| Active governance theme | PDPL aware evidence and zero autonomous action |

## Cohesive interaction model

The user should be able to do four things naturally.

First, switch screens from the sidebar and feel that they remain in the same product world.

Second, switch persona lenses from the top bar and see summaries adapt while the underlying product stays the same.

Third, select a story or governed object and see linked context, architecture, and governance information appear in the main workspace.

Fourth, understand that configuration affects the real operating model, not just abstract settings.

## Visual composition

The layout should feel like a serious product workspace. The left rail is the anchor. The top bar carries context and persona lens controls. The center should act as the main working surface. A contextual side panel can be used inside screens when needed.

The visual language should stay white, blue, and green, but now it should read as product UI rather than marketing. Cards should be denser. Navigation should be clearer. Tables, kanban sections, linked panels, and evidence blocks should dominate more than hero copy.

## Screen priorities

The Command Center should stop being a landing page and become the opening workspace.

The Delivery Flow screen should be the strongest proof that Arkitekt Forge owns the whole governed lifecycle, now including the Operate phase.

The Operate screen should show that Arkitekt Forge does not stop at deployment. Production incidents, service health, and operational lessons are tracked and fed back into the Context Hub.

The Context Hub should show why the product becomes more valuable over time.

The Architecture screen should reveal how the product makes system design reviewable and reusable.

The Governance screen should demonstrate that proof, compliance, and release trust are native.

The Config Studio should show that workflow, gates, memory, and modules are all configurable through UI.
