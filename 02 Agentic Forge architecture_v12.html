Design Choices
Critical Decisions: Why We Built It This Way
Every significant architectural decision is documented here with options considered, rationale, and trade-offs.
Decision
LLM StrategyLLM Abstraction Layer
Wrap Claude Agent SDK in an Execution Service so any LLM provider can be swapped.
Options Considered
1) Call Claude SDK directly (lock-in risk)
2) Build abstraction early (done in v10)
3) Let customers integrate their own (loss of control)
Rationale
Enterprise customers demand multi-provider options. If Claude API changes, only our wrapper updates. Maximizes flexibility without exposing implementation.
Trade-offs
One extra abstraction layer adds ~5ms latency. Worth it for multi-provider portability and customer confidence.
Decision
Data StrategyPostgreSQL as Source of Truth
Single database. No git-sync. nightly pg_dump to GitHub for disaster recovery only.
Options Considered
1) Bi-directional git sync (v3 approach, too complex)
2) PostgreSQL only (chosen)
3) Event sourcing (too much overhead)
Rationale
Git sync was the #1 source of bugs. PostgreSQL with pgvector is simpler, faster, and more reliable. Git is backup, not the source.
Trade-offs
Lose git history as primary record. Gain simplicity and performance. Disaster recovery via pg_dump backups.
Decision
Execution ModelState Machine + 22 States
Kanban-inspired state machine with 22 discrete states across 6 phases (expanded from 17 states and 5 phases in v11).
Options Considered
1) Free-form status field (no governance)
2) 5 states (too simple)
3) 22 states (chosen, granular without being overwhelming)
Rationale
22 states give fine-grained visibility including the new Operate phase. Each state has clear entry/exit criteria.
Trade-offs
More complexity than 5 states. Better governance than fewer. Slightly steeper learning curve for teams, but worth it.
Decision
Agent OrchestrationOrchestrator + Subagents
Parent agent spawns isolated subagents for parallel work within a story.
Options Considered
1) Single monolithic agent (slow)
2) Orchestrator + Subagents (chosen)
3) Fully distributed agents (no coordination)
Rationale
Orchestrator coordinates work. Subagents parallelize reading, writing, testing. Each has isolated context. Orchestrator aggregates results.
Trade-offs
Multiple context windows = more tokens. But speed and parallelism outweigh cost. Simpler coordination than fully distributed.
Decision
Governance ModelSupervised Autonomous Execution
Agents can write and test code in sandboxes. Humans always approve before merge. Updated from Zero Autonomous Action in v11.
Options Considered
1) Full autonomy (unacceptable risk)
2) Supervised autonomous execution (chosen in v12)
3) Zero autonomous action (v11 status quo, too slow)
Rationale
Keeps humans in control of merge decisions. Builds trust with enterprise customers. Clear liability boundaries. Unlocks execution speed.
Trade-offs
Higher token cost (3x to 5x). Requires trust building with developer teams. Developer must understand the explainability report.
Decision
Workflow StrategyConfigurable Workflow Engine
The SDLC is a parameterised engine with customer-defined phases, states, gates, and transitions.
Options Considered
1) Hardcoded 5-phase pipeline (rigid, limits adoption)
2) Configurable workflow engine (chosen)
3) Fully open-ended (no guardrails, risky)
Rationale
Every enterprise has a unique process. A configurable engine adapts to customer workflows rather than forcing them to conform. Default templates provide immediate value. Custom workflows unlock enterprise adoption.
Trade-offs
Significantly more complex runtime than a hardcoded pipeline. Requires validation of arbitrary workflow definitions. Worth it for market breadth and enterprise fit.
Decision
Execution StrategySupervised Execution Over Full Autonomy
v12 introduces a Code Execution Loop where agents autonomously generate, test, and iterate code. But merge requires human approval.
Options Considered
1) Full autonomy (unacceptable liability risk)
2) Supervised execution with human merge approval (chosen)
3) No autonomous execution (v11 status quo, too slow)
Rationale
Enterprise buyers asked two questions: 'Can your agents actually write code?' and 'Who is liable if the code is wrong?' Supervised execution answers both: yes they can write code, and the human who approves the merge accepts responsibility.
Trade-offs
Higher token cost (3x to 5x). Requires trust building with developer teams. Developer must understand the explainability report. Risk of 'rubber stamp' approvals if explainability is poor.

Configurable Workflow Engine
The SDLC Is a Parameterised Engine, Not a Fixed Pipeline
The Arkitekt's workflow engine is fully configurable. Customers define their own phases, states, approval gates, role definitions, and transition rules. The default 6 phases and 22 states are a starting template, not a constraint.
🔧 Why Configurable Over Hardcoded?
Every enterprise has a unique software delivery process. A startup may want three phases. A bank may need twelve. A government agency may require dual approval at every transition. Hardcoding the SDLC would force customers to adapt to the platform. The configurable engine inverts this: the platform adapts to the customer. Default templates provide immediate value on day one, while the full configuration surface unlocks enterprise adoption at scale.
Workflow definitions are stored as JSON schemas in PostgreSQL. The Workflow Definition API allows programmatic creation and modification. The Visual Builder in the UI allows drag-and-drop configuration.

Add, remove, or rename phases beyond the default six. A three-phase startup workflow and a twelve-phase compliance workflow both run on the same engine.
⬜ States
The default 22 states are a template. Each phase can have any number of states with customer-defined names and semantics.

Define how many approvers, which roles must approve, and at which transitions. Single approver for agile teams, dual approval for regulated industries.
👥 Role Definitions
RBAC roles are customer-defined, not platform-imposed. Create custom roles that map to your organisation's structure and permissions model.
🔀 Transition Rules
Entry and exit criteria per state can be customised per project. Define what must be true before a story can move forward or backward.
🏗️ Project Inheritance
Each project can have its own workflow definition, or inherit from an organisation default. Override at the project level without affecting other projects.

Architecture Principles and Governance
The 18 Principles Guiding The Arkitekt
Based on TOGAF, NORA, and NDMO/DAMA frameworks. These principles ensure that governance is not bureaucratic but architectural. v12 adds Principles 17 and 18.
🎯 Supervised Autonomous Execution
Agents execute code in sandboxes. Humans approve merges. Always. Updated from Zero Autonomous Action in v11.
Rationale
Keeps humans in control of merge decisions. Builds customer trust. Clear liability boundaries. Non-negotiable for enterprise. v12 extends this to allow supervised code execution within governed boundaries.
Impact
Faster than zero autonomous action, safer than full autonomy. Requires human attention for merge approvals. Trade-off: speed with safety.
Enforcement
Blocking Control #1: No agent can merge to main without human approval. Agents can write and test code in sandboxes. Enforced at API boundary and Governance Pipeline.
🚪 Human Governance at Every Gate
Six phases. Six human approval gates. No shortcuts.
Rationale
SDLC gates ensure quality at each step. Plan review prevents wasted effort. Design review catches architecture issues early.
Impact
Adds ~2 to 5 hours of human time per story. Early feedback prevents rework. Net savings in overall timeline.
Enforcement
State Machine enforces transition rules. BullMQ job queue blocks progression until human approves.
📋 Immutable Audit Trail
Every decision recorded. Hash-chain proof. No editing history.
Rationale
Regulatory compliance (SOX, HIPAA). Dispute resolution. Root cause analysis. Legal defense.
Impact
Adds ~10% storage overhead. No negative performance impact. Massive value for regulated industries.
Enforcement
Proof Chain immutability enforced at database layer. PostgreSQL constraints. Git commits embedded in PR descriptions.
🔐 Least Privilege for Agents
Agents get exactly the permissions they need. Nothing more.
Rationale
Security principle. Limits blast radius if agent is compromised or behaves unexpectedly.
Impact
Agents need to ask for elevated permissions. Slight overhead in tool design. Massive security benefit.
Enforcement
Tool visibility enforced at API level. Persona SHA-256 check. GitHub token scopes. Vault path prefixes.
📊 Complete Context Compilation
Agents get 8-step brief before every execution. Full codebase context via pgvector.
Rationale
Better decisions require better context. Memory nodes compound per customer. Agents never start cold.
Impact
Context compilation takes ~2 to 3 seconds. Offset by better code quality and fewer rework cycles.
Enforcement
Context Compiler mandatory before agent execution. Gating at BullMQ job level. Metrics tracked.
🔄 Transparent Decision Making
Agents explain their reasoning. Humans can understand and challenge.
Rationale
Black-box AI is unacceptable in enterprise. Explainability builds trust. Enables human override.
Impact
Agents provide thought process. Humans can trace logic. Sometimes requires human intervention.
Enforcement
Agent logs stored in PostgreSQL. Displayed in UI. Accessible in API response.
🎚️ LLM Provider Abstraction
Execution Service wraps any LLM. Customers never see the provider.
Rationale
Avoids vendor lock-in. Enables model upgrades without API changes. Customer selects in Settings.
Impact
Small abstraction overhead (~5ms). Worth it for portability and customer autonomy.
Enforcement
Execution Service is the single point of LLM integration. All model calls flow through here.
🌐 API-First Design
Everything flows through the REST API. UI is just a client.
Rationale
Enables SDK embedding. Powers integrations. Clear separation of concerns. Easier to scale.
Impact
Frontend must make API calls like any third-party client. Slight latency added. Massive architectural benefit.
Enforcement
Hono API server is the authority. UI calls only via REST + WebSocket. RBAC enforced at API layer.
🧪 Proof Chain Verification
Hash-chain validates context, directive, and artifacts at every step.
Rationale
Prevents tampering. Documents what was used to make each decision. Enables audit and compliance.
Impact
GitHub Actions verifies proof chain before human review. ~500ms added per merge. Standard practice.
Enforcement
CI/CD pipeline validates proof chain. No merge without proof. Embedded in PR description.
🚨 Governance Blocking Gates
10 mandatory gates + 3 advisory signals. Examples: hallucination detection, PII scanning, security validation.
What Gets Blocked
PII in code, hallucinated facts, insecure API calls, incomplete tests, code smells.
Advisory Signals
Giskard: hallucination score, prompt injection risk, sycophancy likelihood.
Enforcement
Governance Pipeline runs after CI passes. Blocks merge unless all gates pass. Human can override with justification.
💾 Single Source of Truth
PostgreSQL is the authority. Not git. Not a cache.
Rationale
Eliminates sync drift. Simpler than bi-directional replication. nightly pg_dump to git for disaster recovery.
Impact
No git-sync complexity. PostgreSQL as primary record. Trade-off: lose git history as source.
Enforcement
All state changes write to PostgreSQL first. API queries PostgreSQL. Redis caches for performance.
📈 Observability by Default
Every agent call logged. Token usage tracked. Cost metered. Latency measured.
Rationale
Enables cost control. Performance optimization. Troubleshooting. Fraud detection.
Impact
~5 to 10% overhead for logging. Visible in dashboards. Cost transparency builds customer trust.
Enforcement
Logging middleware in API layer. Agent SDK integration. Real-time metrics in UI.
🔌 Infrastructure Abstraction
All infrastructure swappable. API contracts never change.
Rationale
Upgrade without disruption. Try new technologies. Optimize infrastructure without touching API.
Impact
Clean API boundaries. Slightly thicker abstraction layers. Worth it for long-term flexibility.
Enforcement
Execution Service wraps LLM. Job queue is BullMQ today, swappable tomorrow. Database is PostgreSQL, swappable later.
☁️ Cloud Native, Cloud Agnostic
Requires Kubernetes, PostgreSQL, Redis, S3-compatible storage. Does not require any specific cloud provider.
Rationale
Enterprise customers operate on different cloud providers. Government customers may require on-premises deployment. The platform must run anywhere Kubernetes runs.
Impact
No proprietary cloud services in the critical path. All dependencies expressed as abstract contracts (e.g., "S3-compatible" not "AWS S3"). Slightly more operational complexity.
Enforcement
Cloud Abstraction Contract defines every infrastructure dependency as an interface. CI validates that no cloud-specific APIs are called directly.
🏢 Tenant Isolation Scales with Sensitivity
Data boundaries enforced architecturally, not through application logic alone.
Rationale
SaaS customers need cost-effective isolation. Regulated enterprises need database-level separation. Government needs full infrastructure isolation. One model does not fit all.
Impact
Three isolation tiers with different cost and security profiles. Customers select the tier that matches their regulatory requirements. Operational complexity increases with tier.
Enforcement
Tier 1: Row Level Security in PostgreSQL. Tier 2: Dedicated database instances. Tier 3: Dedicated Kubernetes clusters. Enforced at infrastructure provisioning.
🔧 Configurable Workflow, Not Hardcoded Process
The SDLC is a parameterised engine, not a fixed pipeline. Customers own their process definition.
Rationale
Enterprise software processes vary widely. A startup needs agility. A bank needs governance. A government agency needs compliance. The workflow engine adapts to all of them.
Impact
Workflow definitions are JSON schemas. Default templates ship out of the box. Customers can modify phases, states, gates, and transition rules. Each project can override the org default.
Enforcement
Workflow Definition API validates schemas before activation. Visual Builder provides guardrails. Runtime enforces whatever the customer defines.
🚀 Supervised Autonomous Execution
Agents can write and test code inside isolated sandboxes. Only humans can approve merges. Autonomy level configurable per workflow.
Rationale
Closes the gap between governance and execution without sacrificing trust. Enterprise customers maintain control. The level of autonomy scales with organisational maturity.
Impact
Faster delivery (est. 40 to 60% reduction in cycle time). Requires developer trust building. Token cost increases 3x to 5x per story.
Enforcement
Workflow Engine controls autonomy level per template. Maximum iteration count enforced at sandbox level. Automatic escalation on repeated failures.
💰 Context Budget Awareness
Every agent execution operates within a hard token budget. Context Compiler prioritises relevance within the ceiling.
Rationale
Context windows have practical limits even at 1M tokens. Without budget awareness, context quality degrades silently. Budget allocation ensures the most critical information always fits.
Impact
Prevents silent quality degradation. Adds ~200ms to context compilation. Enables cost predictability for metering.
Enforcement
Context Budget Manager calculates allocation before every execution. Hard ceiling enforced at Model Router. Overflow triggers prioritisation algorithm.

Modular Agent and Extension System
Plug and Play Module Architecture
The Arkitekt uses a modular architecture where specialised agents, tools, governance gates, and language runtimes can be added or removed per project without platform restart. Each module is versioned, containerised, and hot-swappable.
🧩 Module System Architecture
The Module Registry is a versioned catalogue of all available modules. Each module declares its capabilities, dependencies, and governance requirements through a manifest. Modules are deployed as container images and can be added to any project at runtime. The framework supports 14 native module categories, 9 configurable categories, and 12 add-on categories from the use case roadmap.
Hot-swappable: adding a new agent module does not require platform restart. The Module Registry handles discovery, versioning, and deployment lifecycle.
Module Categories

Agent Modules
Specialised subagents that can be added or removed per project. Each agent module has domain expertise and a defined tool surface.
Examples: Mobile Developer Agent, Data Engineer Agent, Infrastructure Agent, E2E Test Agent, Security Audit Agent, API Design Agent

Pluggable tools with bidirectional capabilities that agents can use. Each tool has a manifest, a runtime, and governance metadata defining its permissions and audit requirements.

Code Execution Loop
From Approved Design to Governed Pull Request
The Code Execution Loop closes the critical gap between governance and code delivery. It takes approved design artifacts and autonomously generates, tests, and iterates code within governed boundaries. Humans review and approve the final output. New in v12.
🛡️ Principle Update: Supervised Autonomous Execution
v11 enforced Zero Autonomous Action as a blanket rule. v12 introduces Supervised Autonomous Execution: agents can write and test code inside isolated sandboxes, but only humans can approve the merge. This preserves governance while unlocking execution speed. The autonomy level is configurable per workflow template.
1
Design Artifact Approved
Human
Human approves the design at State 6 (Ready Dev). The approved design artifact, architecture decisions, and Context Hub patterns are assembled by the Context Compiler into an Executable Specification.
2
Context Budget Allocation
System
Context Budget Manager calculates the optimal context window allocation. Prioritises: design artifact (40%), codebase understanding (25%), related patterns (20%), governance rules (15%). Hard ceiling prevents context overflow.
3
Sandbox Code Generation
Agent
Coding subagent spawns inside OpenSandbox. Generates code following the Executable Specification. Writes to feature branch. Each file change is traced in proof chain with reasoning explanation.
4
CI/CD Validation
System
GitHub Actions triggered automatically. Runs unit tests, integration tests, linting, and security scans. CI results are read back into the agent context. SonarQube, CodeQL, and Snyk results feed into Governance Pipeline.
5
Iteration or Escalation
Agent
If CI fails: agent reads error output, diagnoses root cause, and generates fixes (maximum 5 iterations). If iteration limit exceeded: automatic escalation to human developer with full context and failure history.
6
Explainability Report
System
Agent Explainability Layer generates PR documentation: reasoning chain, Context Hub patterns referenced, design artifact followed, governance gates passed, iteration history, and token consumption. Designed for developer consumption, not just audit.
7
Human Review and Merge
Human
Developer reviews the PR with full explainability report. Can approve, request changes, or reject. Only after human approval does the code merge. Governance Pipeline validates proof chain before merge is permitted.

Integration, Not Replacement

Forge consumes telemetry from external tools (Datadog, Grafana, CloudWatch, Azure Monitor, New Relic) through Tool Connectors (Layer 8). The monitoring tools detect the problem. Forge governs the response: who approves the rollback, what evidence is captured, what the root cause analysis says, how the lesson is stored in Context Hub.

This avoids competing with established observability platforms while owning the governance narrative. Forge adds a governed workflow layer on top of monitoring signals, turning alerts into governed incident stories with evidence chains, approval gates, and operational memory.

Key distinction: Monitoring tools answer "what happened." Forge governs "what we do about it" and "what we learn from it."

Multi-Tenancy and Tenant Isolation
Three Isolation Tiers for Every Customer Profile
Tenant isolation scales with data sensitivity. SaaS customers get cost-effective logical isolation. Regulated enterprises get dedicated databases. Government gets full infrastructure separation. All three tiers run on the same platform codebase.
Tier 1
Logical Isolation
Single shared infrastructure with strong logical boundaries. Cost-effective for SaaS customers who need standard data protection without dedicated resources.
PostgreSQL: Single cluster, Row Level Security (RLS) per tenant
Redis: Keyspace prefix isolation per tenant
S3: Path partition per tenant (tenant_id/...)
Vault: Shared namespace with path-based access control
Kubernetes: Shared cluster, namespace isolation

Best for: Startups, small teams, standard SaaS customers
Tier 2
Database Isolation
Shared compute with dedicated data stores. For regulated enterprises that need stronger data boundaries without the cost of full infrastructure isolation.
PostgreSQL: Dedicated instance per tenant
Redis: Dedicated Redis instance per tenant
S3: Dedicated bucket per tenant
Vault: Dedicated namespace per tenant
Kubernetes: Shared cluster, dedicated namespaces

Best for: Regulated enterprises, financial institutions, healthcare
Tier 3
Full Infrastructure Isolation
Completely dedicated deployment. Every component runs in isolation. For government agencies and organisations with the highest security and compliance requirements.
PostgreSQL: Dedicated cluster
Redis: Dedicated cluster
S3: Dedicated storage account
Vault: Dedicated Vault instance
Kubernetes: Dedicated cluster
Network: Dedicated VPC/VNet

Best for: Government, military, critical national infrastructure
Cloud Native, Cloud Agnostic Deployment

Managed Kubernetes
Deploy on any managed Kubernetes service from a major hyperscaler. Lowest operational overhead. The platform team manages the application; the cloud provider manages the infrastructure. Supports GKE, EKS, AKS, or any CNCF conformant managed offering.

API Versioning and SDK Lifecycle
Stable APIs. Predictable Deprecation. Semantic Versioning.
The Arkitekt follows strict API versioning and SDK lifecycle policies to ensure customers can build confidently on the platform without fear of breaking changes.
📡 URL-Based Versioning
All API endpoints are versioned in the URL path: /api/v1/, /api/v2/, etc. Major version increments indicate breaking changes. Minor and patch changes are backward-compatible within a major version. All versions run concurrently during overlap periods.
📅 Version Lifecycle
Each API major version is supported for a minimum of 18 months after the next major version is released. Deprecation notices are published 6 months before end of support. Customers receive automated migration guides and API diff reports when a new major version is available.
📦 SDK Semantic Versioning
TypeScript and Python SDKs follow semantic versioning (SemVer). A compatibility matrix maps SDK versions to supported API versions. Breaking SDK changes increment the major version. SDKs include TypeScript type definitions and Python type hints for IDE support.
📝 Machine-Readable Changelogs
Every API version publishes an OpenAPI spec. Differences between versions are available as machine-readable OpenAPI diffs. CI/CD pipelines can validate client compatibility against the latest API spec. SDK changelogs follow Keep a Changelog format.
